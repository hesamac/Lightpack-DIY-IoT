## 1. Zone → LED socket remap

The firmware zone indices didn't line up with the socket numbers printed on the
enclosure. Measured per-socket on the board: the near DM631 (sockets 1–5) is
wired in a scrambled order; the far DM631 (sockets 6–10) is sequential.

Fix: a one-line indirection table in `dm631.c` so logical zone N drives box
socket N+1. `build_frame()` and the Matter layer are untouched.

```c
// Which physical DM631 zone each LED socket is wired to (LED 1 = index 0):
//   LED 1 → Zone 4     LED 6  → Zone 5
//   LED 2 → Zone 3     LED 7  → Zone 6
//   LED 3 → Zone 0     LED 8  → Zone 7
//   LED 4 → Zone 1     LED 9  → Zone 8
//   LED 5 → Zone 2     LED 10 → Zone 9
static const uint8_t led_to_zone[DM631_NUM_ZONES] = {4, 3, 0, 1, 2, 5, 6, 7, 8, 9};
```

Tiles were also renamed "Zone N" → "LED N" (NodeLabel; shows on next pairing).

## 2. HS-native simplification (fixed the multi-light storm)

**Symptom (Issue 2):** turning all 10 lights on/off together, or asking Siri to
set a colour, produced 2–3 s of chaotic flicker before settling.

**Root cause:** an over-engineered colour path. A "mirror" system wrote every
colour change across all three representations (HS↔XY↔CT) on every command, and
ColorControl was handled in *two* callbacks. A single `MoveToHueAndSaturation`
fanned out into ~8 LED applies plus a flood of outbound attribute reports — and
with 10 zones at once it overwhelmed the Matter pipeline.

**Diagnosis aid:** confirmed via an on-device command trace that Apple Home
drives colour exclusively with `MoveToHueAndSaturation` (command `0x06`) — i.e.
the device is **HueSaturation-native**, so the XY-forcing/mirroring was fighting
Apple.

**Fix:** drive the LED from HS as the single source of truth; handle
ColorControl in exactly one place (`MatterPostAttributeChangeCallback`, which
fires *after* the value is stored so it always reads fresh). Deleted the mirror,
the boot HS→XY normalise, the `ScheduleWork` trampoline, the re-entry guard, the
mismatched RGB↔XY matrices, and the stale-reading PRE_UPDATE branch.
Result: one command → one apply → one DM631 frame. **−168 lines.** Storm gone.

## 4. The light-blue slider — the long one

**Symptom:** set a light to a colour → the LED is correct, but on reopening the
Home app the colour swatch/slider shows a default **light blue**. Persisted
across many earlier fix attempts.

### How we finally pinned it

1. **Instrumented the device** with the chip shell
   (`matter esp attribute get <ep> 0x300 <attr>`). With a zone set to green the
   device held *perfect, consistent* green: ColorMode=0 (HS), CurrentHue=84,
   CurrentSat=254, CurrentX/Y = green, CT = 444 mireds (warm). **None** of the
   representations is light blue → Apple was showing its **generic default
   swatch**, i.e. not applying our reported colour at all on a fresh read.
2. **A/B tested the topology.** Rebuilt non-bridged (one composed accessory
   instead of the Aggregator + bridged lights). Still light blue — *and* it lost
   tap-to-toggle. So the bridge was **not** the cause (and is strictly better:
   separate tiles + tap-to-toggle at no extra cost). Bridge restored.
3. **Web search** found the real cause: a widely-reported **Apple-side bug** —
   HomeKit mis-handles lights that advertise **Color Temperature alongside
   Hue/Saturation**, painting the swatch a default white/blue on a fresh read.
   It even affects CT-exposing Philips Hue bulbs. (Hubitat / Home-Assistant /
   Hue community threads all describe the exact symptom.)

### The fix

Advertise **HS + XY only** — drop Color Temperature from `ColorCapabilities`
and `FeatureMap` (`0x19` → `0x09`, CT bit 4 cleared) in
`configure_color_endpoint()`. Apple then treats the device as a pure colour
light and reads the real hue/saturation. **Confirmed fixed.**

Trade-off: no "Temperature" tab in the Apple colour picker — irrelevant for an
RGB device. And because it's topology-independent, we kept the bridge (separate
tap-to-toggle tiles) *and* got correct colour.

## Lessons / gotchas

- **Apple Home is HueSaturation-native** for colour control — don't force XY/CT.
- **Don't advertise Color Temperature on an RGB-only light** for Apple Home, or
  the swatch renders as a default white/blue on fresh reads.
- **Instrument before guessing.** The chip-shell attribute reads (and the
  command trace) ended ~5 cycles of guess-and-flash by proving the device state
  was correct and the issue was Apple-side.
- **A/B test structural hypotheses** before big rewrites — it disproved the
  bridge theory cheaply and redirected us to the real cause.
- Serial port name varies with the USB socket; a stale `idf.py monitor` keeps
  the port locked (`lsof -t /dev/cu.usbserial-XXX | xargs kill`).
