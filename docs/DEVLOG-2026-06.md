# Lightpack DIY — Dev Log (June 2026)

Replacing the original **AT90USB162** microcontroller on a Lightpack rev6.0L
board with an **ESP32-C6** to expose the 10 RGB LED zones as Apple HomeKit
(Matter) lights.

**Result of this week: fully working end-to-end.** All 10 zones drive correct
colour and brightness from Apple Home; all known UI bugs resolved.

---

## Hardware

### Board topology
- **IC1** = AT90USB162 (original MCU, being retired)
- **IC2** = DM631 LED driver, near chip, zones 0–4
- **IC3** = DM631 LED driver, far chip, zones 5–9 (daisy-chained after IC2)
- 3-wire serial bus shared by both DM631s: **CLK, DATA, LATCH**

### ESP32-C6 wiring
| ESP32-C6 | Signal | Board point |
|----------|--------|-------------|
| GPIO6  | CLK    | IC1 pin 15 / IC2 DCK |
| GPIO7  | DATA1  | IC1 pin 16 / IC2 DAI |
| GPIO14 | LATCH  | IC1 pin 14 / IC2 /LAT |
| GND    | GND    | any board ground |

**Do NOT connect ESP32 3.3V/5V to the board.** Three signal wires + GND only.
All supplies must share a common ground.

### Power (the key hardware lesson)
- **12V** on the power jack feeds **only the LED anode rail** (isolated by diode
  D2/SS14). It does **not** power the DM631 logic.
- **DM631 VDD (pin 24) = +5V**, which on this board comes **only from the
  board's USB connector**. With the board powered from 12V alone, VDD read 0V
  and the chips ran on input-pin leakage → chaotic, meaningless LED output.
- **Fix:** feed 5V into the board's USB port (a phone charger, or the project's
  MP1584 buck — VBUS + GND only, leave D+/D- unconnected). Confirmed: VDD = 5.22V.
- LED rail correctly stays at 12V (the strips are 12V type).

### DM631 protocol (verified)
- Frame: `[0pad][z5..z9]` then `[0pad][z0..z4]`, 32 × 12-bit = 384 bits, MSB-first,
  LATCH pulse after.
- **Channel order per zone: B → G → R** (matches the original woodenshark
  `Lightpack-master/Firmware/LedDriver.c`; the `at90usb162-firmware` copy's
  B→R→G is wrong).
- Runs reliably at full ESP32 GPIO speed (no added bit delays) now that VDD = 5V.
- Logic-level note: DM631 VIH(min) = 3.5V vs ESP32's 3.3V is marginally out of
  spec but works in practice. If it ever glitches: drop the logic rail to ~4.5V
  or add a 74AHCT125 buffer.

### Retiring IC1
- IC1 was held off by grounding its RESET pin (pins go Hi-Z → no bus contention).
- RESET has a 20kΩ pull-up to +5V, so a broken ground tie lets IC1 wake and
  fight the bus — verify RESET reads ~0V if in doubt.
- Cleanest permanent fix: cut IC1's leads at the package body (at minimum pins
  14/15/16) — the PCB pads/traces and the ESP32→DM631 path survive; leftover
  pin stubs are inert.

## Firmware

ESP-IDF + esp-matter project in `firmware/`. Standalone hardware test in
`dm631_test/` (no Matter — all-white, per-zone RGB sweep, brightness ramp,
rainbow).

### Matter model
- **Bridge topology:** EP1 = Aggregator, EP2–EP11 = bridged Extended Colour
  Lights named "Zone 1".."Zone 10". Each becomes a first-class HomeKit accessory
  with tap-to-toggle, its own room, and independent scenes/automations.
- `CONFIG_ESP_MATTER_MAX_DYNAMIC_ENDPOINT_COUNT = 12` (root + aggregator + 10).
- ColorControl advertises HS + XY + CT (FeatureMap / ColorCapabilities = 0x19),
  set manually after endpoint creation due to an esp-matter ODR FeatureMap bug.

### Colour pipeline
- HSV/CT/XY → RGB, **gamma 2.2**, 12-bit per channel to the DM631.
- Colour changes are **mirrored across HS/XY/CT** so any controller readback is
  consistent (with a re-entry guard so mirror writes don't recurse).
- Zones **normalise to XY colour mode at boot** — Apple Home renders the slider
  only from XY/CT; an HS-mode zone showed a generic light-blue fill.

## Bugs fixed this week

| Symptom | Root cause | Fix |
|---------|-----------|-----|
| Chaotic LEDs / no white | DM631 logic unpowered (VDD = 0V) | Feed 5V to board USB port |
| Wrong colours | Channel order | Confirmed B→G→R |
| Colours washed out | No gamma | Gamma 2.2 |
| Slider "light blue", LED correct | Stale CT attr; HS mode not rendered by Apple | Advertise CT; mirror attrs; boot-normalise HS→XY |
| Sub-second wrong-colour flash | LED applied on ColorMode write before values | Skip mode/RemainingTime writes in callback |
| Brightness snaps to 25% on every toggle | OnLevel = 64 (spec forces it) | OnLevel + StartUpCurrentLevel = null |
| Zones show "off" while LED on | Wi-Fi modem power-save latency | esp_wifi_set_ps(WIFI_PS_NONE) |
| Laggy UI / loading spinner | KBs of CHIP logs per command on Matter thread | Power-save off + CHIP log tags → WARN |
| Can't tap-toggle from tile icon | Composed device, not a bridge | Bridge (Aggregator + bridged nodes) |

### Known / by-design
- Colour transitions step rather than glide — inherent to Matter-over-Wi-Fi
  (~30–50ms RTT) vs Hue's Zigbee; not fixable in firmware.
- Device advertises two operational identities → a stale fabric remains in NVS;
  an `erase-flash` + re-pair cleans it if desired.

## Build / flash / monitor

```bash
# Every new terminal:
source ~/esp/esp-idf/export.sh
source ~/esp/esp-matter/export.sh   # only needed for build (sets ESP_MATTER_PATH)

# Main firmware:
cd "firmware" && idf.py -p /dev/cu.usbserial-XXX build flash monitor

# Standalone hardware test:
cd "dm631_test" && idf.py set-target esp32c6 && idf.py -p /dev/cu.usbserial-XXX build flash monitor
```

- Serial port name varies with the USB socket (`/dev/cu.usbserial-210`,
  `-110`, …) — check `ls /dev/cu.*`. If "port busy", an old monitor is still
  open: `lsof /dev/cu.usbserial-XXX` then kill it.
- Apple Home pairing code: **20202021**.
- The repo-root `CMakeLists.txt` is a junk "404" placeholder; the real project
  is in `firmware/`. Always run `idf.py` from `firmware/` (or `dm631_test/`).
- Re-pairing is required after the bridge restructure (Remove Accessory →
  `erase-flash` → re-pair).
